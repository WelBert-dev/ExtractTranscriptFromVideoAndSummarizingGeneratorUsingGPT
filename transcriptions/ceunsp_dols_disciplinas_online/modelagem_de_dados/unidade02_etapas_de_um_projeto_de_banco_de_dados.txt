Vamos falar agora sobre as etapas de um projeto de banco de dados e o conceito de modelagem, além do levantamento de requisitos na regra de negócios e o processo de desenvolvimento lógico para o banco de dados. Vem comigo! Desenvolver sistemas é uma atividade complexa e a modelagem de banco de dados é um passo essencial no processo de criação de qualquer software. No centro da modelagem reside a ideia de projetar uma estrutura de banco de dados que define como as informações armazenadas podem ser acessadas, categorizadas e manipuladas. Essa definição é fundamental para que a recuperação do dado possa ser feita de forma íntegra. Assim, a modelagem de dados tem diversos objetivos. O primeiro deles é identificar os diferentes componentes de dados que são relacionados ao tipo de informação que vai ser armazenada. Esse processo deve considerar os dados brutos processados, bem como as entidades, ou seja, diferenciar o que não precisa ser armazenado referente a um processamento. Por exemplo, não precisamos guardar o campo idade, já que ele é proveniente de um cálculo da data de nascimento. A modelagem deve reconhecer também as relações entre diferentes componentes de dados, que são chamadas de associações. Isto é, o relacionamento entre as tabelas. Além disso, a modelagem identifica os usos antecipados dos dados, chamados de requisitos, sabendo que os dados podem ser mais valiosos no futuro para usos não previstos. A modelagem reconhece ainda os pontos fortes e as restrições das tecnologias que se planeja usar no projeto. Esse objetivo está associado ao hardware e ao hardware que vem a ser utilizado. Então, quando fazemos essa análise, é importante reconhecer qual plataforma vai ser executada para que se possa ter o melhor aproveitamento daquele banco. Por fim, a modelagem visa construir um modelo preliminar das entidades e as suas relações, tentando manter o modelo independente de quaisquer usos específicos ou restrições de tecnologia. Devemos destacar que o cliente que acessa a informação não precisa saber detalhes relacionados ao banco utilizado. Cabe ao arquiteto do processo de levantamento tornar isso o mais transparente possível para que o desenvolvedor acesse a informação, retorne e exiba ao cliente final. Assim, modelos de dados bem documentados permitem que os interessados identifiquem erros e façam alterações antes de qualquer código de programação ter sido escrito. Para entendermos os modelos de dados, temos que conhecer algumas diferenças entre dados, informação e conhecimento. Quando falamos em dados, eles ainda estão em sua forma bruta e não permitem a tomada de decisão sem a análise prévia. Por exemplo, podemos pensar na quantidade de nascimentos dos últimos 10 anos, que é um dado, mas por si só não temos essa informação. Então, quando falamos em dados, indicamos cada item e armazenato dentro do banco. A partir da análise, eu tenho a informação que pode influenciar a tomada de decisão. Assim, temos o conhecimento sobre o assunto. O levantamento de requisitos é uma etapa muito importante desse processo, pois é quando vamos definir as regras de negócio. Requisitos são os artefatos que determinam o que o sistema deve definir. Já regras sempre vão existir e definir a guarda e a qualidade da informação. Por exemplo, é obrigatório coletar o nome do cliente? Ou ainda, a data de nascimento deve ser superior a 1900? Então, o objetivo desse levantamento de requisitos é identificar a situação do mundo real junto com o cliente, para que seja possível delimitar esses componentes no banco de dados. De forma geral, temos dois tipos de dados, os dados naturais, que são a entrada e os processados ou de saída. Se pensarmos em um exemplo de faturamento mensal, os dados de entrada representam a coleta de faturamento, são os dados brutos. Então, é realizado o processamento com a somatória do que foi executado no mês, para identificar o faturamento mensal. Nessa etapa de levantamento, o analista responsável deve buscar delinear os requisitos e dados da empresa em termos de elementos dos dados básicos, descrever as informações sobre os elementos dos dados e as relações entre eles necessárias para modelar esses requisitos e determinar os tipos de transações que pretendem executar. Por exemplo, desenvolvemos um sistema para a venda de ingresso que vai experimentar um grande volume de acesso por causa de um evento e precisamos identificar quantas transações podem ser executadas simultaneamente. Então, temos que garantir que aquela pessoa que reservou o ingresso e fez o pagamento tenha seu lugar reservado por completo. Do contrário, se a operação não for finalizada, temos que desfazer e disponibilizar essa vaga novamente no banco de dados. O analista também deve buscar definir quaisquer restrições de desempenho, integridade, segurança ou administrativas que devem ser impostas ao banco de dados resultante. Isso porque as informações da empresa devem ser armazenadas de uma maneira azeleosa para prevenir invasões. Além disso, o analista deve especificar quaisquer restrições de projeto e de implantação, como tecnologias, hardware e software, linguagens de programação, políticas, padrões e interfaces externas. As regras de negócios são uma declaração que define ou restringe o acesso. Elas permitem que o analista desenvolva normas e restrições de entidades e relacionamentos com os interessos e crie um modelo de dados condizente com o mundo real. Por exemplo, uma pessoa faz um cadastro no meu sistema e troca de endereço. Podemos optar por guardar o histórico dessa alteração, se julgarmos necessário armazenar esse dado para uma análise futura. Esse é um exemplo de regra de negócio. Por isso, o levantamento e a regra de negócio seguem sempre em paralelo. Agora, vamos falar sobre os níveis de modelagem. O primeiro nível é do modelo conceitual. Nessa etapa, ainda não foi definido o tipo de banco de dados que vai ser utilizado. O modelo conceitual identifica as relações de nível mais alto entre as diferentes entidades, sem especificar nenhum atributo ou chave primária. Por exemplo, vamos montar um modelo conceitual. Um proprietário pode comprar um produto e levar um animal para uma consulta no veterinário. Percebam que não especificamos que tipo de banco de dados estamos utilizando. Já o segundo nível corresponde ao modelo lógico, que descreve os dados com o máximo de detalhamento possível, independentemente do modo como vai ser sua implementação física. Nesta fase, identificamos as entidades e relações entre elas, os atributos e a chave primária para cada entidade e as chaves estrangeiras. Além disso, a normalização ocorre neste nível. No exemplo em tela, estamos utilizando o modelo relacional. Então, eu tenho seis tabelas. Propretário, compra, produto, animal, consulta e veterinário. O proprietário quer comprar um determinado produto, e o animal passa pelo veterinário por meio de uma consulta. Portanto, nesta etapa, montamos os relacionamentos e identificamos o tipo de banco de dados a ser utilizado. Por fim, no terceiro nível, temos o modelo físico, que representa como o projeto vai ser construído no banco de dados. Ele mostra todas as estruturas de tabela, incluindo nome, tipo de dados e restrições da coluna, além da chave primária, chave externa e relações entre tabelas. Nesta etapa, pegamos a tabela e aplicamos a linguagem S e S. Ou seja, escolhemos exatamente a ferramenta que vamos utilizar, como o Oracle, o MySQL ou o Firebird. Com isso, abordamos o conceito da modelagem de dados e seus níveis de abstração. Falamos sobre a importância de levantamento de requisitos e da regra de negócio para processamento e desenvolvimento do projeto de brilho e de fio. Bom, os estudos e até a próxima!
